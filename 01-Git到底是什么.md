# 第一课：Git 到底是什么？

这一课其实非常关键，因为你最开始问的：

> Codex 的工作区域怎么定比较好？是在云端还是在本地？

真正涉及的核心概念，就是 Git。

学完这一课，你会明白：

* Git 是什么
* Commit 是什么
* Repo 是什么
* Branch 是什么
* PR 是什么
* 为什么 AI Coding 工具都在使用 Git

---

## 你一定经历过这种情况

电脑里出现：

```text
商业计划书.docx

商业计划书_v2.docx

商业计划书_v3.docx

商业计划书_最终版.docx

商业计划书_最终版2.docx

商业计划书_最终版_真的最终版.docx
```

过了三个月：

你已经不知道哪个是最新的。

---

这其实就是 Git 要解决的问题。

## Git 的本质

一句话：

> Git 是一个记录文件历史变化的系统。

它不是编程工具。

它首先是一个：

> 超级版本管理工具。

---

## 先理解一个最重要的概念

### 时间机器

假设你每天写一点书。

第一天：

```text
版本A
```

第二天：

```text
版本B
```

第三天：

```text
版本C
```

第四天：

```text
版本D
```

普通文件夹：

```text
A
B
C
D
```

你需要自己保存。

---

Git：

```text
A → B → C → D
```

全部自动记录。

并且随时可以说：

> 给我看看上周二那个版本。

甚至：

> 回到上周二那个版本。

---

这就是 Git 最核心的价值。

---

# Commit（提交）

这是 Git 世界最重要的词。

很多人被翻译害了。

其实不要理解成「提交」。

理解成：

> 拍快照。

---

例如：

上午：

```text
商业计划书
```

你写了两小时。

觉得：

> 这一版不错。

于是拍张照片。

```text
快照1
```

---

下午又改。

再拍一张：

```text
快照2
```

---

晚上再改。

再拍：

```text
快照3
```

---

Git 里的 Commit：

```text
快照1

快照2

快照3
```

就是这个意思。

---

所以以后看到：

```text
Commit
```

脑子自动翻译成：

> 存档点

或者：

> 快照

就行了。

---

# Repository（Repo）

第二个最常见的词。

很多 AI 工具天天提。

例如：

* GitHub Repo
* Open Repo
* Clone Repo

---

其实很简单。

### Repo = 项目档案柜

想象一个项目：

```text
项目
├── 文档
├── 图片
├── 表格
├── 配置
└── 历史记录
```

整个项目放在一起。

这就是 Repo。

---

以后看到：

```text
Repository
```

直接翻译：

> 项目仓库

或者：

> 项目档案柜

---

# Branch（分支）

这是最容易让非技术人士突然开窍的概念。

---

假设你有一本正式合同。

老板说：

> 我想试试修改条款。

怎么办？

直接改正式版？

风险太大。

---

正确做法：

复印一份。

在复印件上改。

---

正式版：

```text
合同A
```

实验版：

```text
合同A-修改版
```

---

如果修改失败：

```text
扔掉复印件
```

正式版没受影响。

---

这就是 Branch。

一句话：

> Branch = 正式稿的副本。

---

以后看到：

```text
Create Branch
```

直接翻译：

> 复制一份出来试验。

---

# 为什么 AI 都喜欢开 Branch？

现在回到 Codex。

假设你说：

> 帮我修改网站。

Codex 有两个选择：

### 方案1

直接修改正式版

```text
正式版
↓
AI乱改
```

风险极大。

---

### 方案2

先复制

```text
正式版
↓
Branch
↓
AI修改
```

失败：

```text
删除 Branch
```

结束。

---

所以几乎所有 AI Coding 工具都会优先：

```text
开 Branch
↓
修改
↓
测试
↓
提交结果
```

---

# Pull Request（PR）

最后一个概念。

很多 AI 都会说：

> 我已经创建 PR。

---

PR 本质上是什么？

其实就是：

> 修改申请。

---

例如：

你把合同修改好了。

交给老板：

> 我建议改成这样。

老板看完：

* 同意
* 不同意
* 继续修改

---

PR 就是这个过程。

---

所以：

```text
Branch
↓
修改
↓
PR
↓
审核
↓
通过
```

这才是标准流程。
